# 成本管理系统前端改造方案

> 目标：最终完成 V3 独立化，但这次改造的主目标不是“拆更多文件”，而是**减少代码数量、提高代码质量、降低阅读和排错成本、让模块职责清晰且不串台**。  
> 原则：**每次只动一个功能模块，先保行为不变，再做结构收口；不靠兜底救火，优先在源头收口问题。**

## 零、评审结论与执行纠偏

### 0.1 总体判断

这份方案可以继续用，但必须先调整主导思想。  
当前版本更像一份“拆文件施工单”，而你现在要的目标其实更高一层，是一份“**减量 + 分层 + 清边界 + 缩短排错链路**”的重构方案。

原方案里下面 4 点仍然成立：

- 先做低风险清理，再拆大文件，最后动核心状态层，这个顺序总体合理
- 把 `useMetaConfig`、`runtime/index.ts`、`TablePanel.vue` 作为大文件热点识别出来，是准确的
- 把“状态所有权不清、缓存更新后 UI 不同步”定义为核心问题，也是准确的
- 强调“每一步独立可验证”，这个原则必须保留

但这份方案**不能再以“把大文件拆成小文件”作为成功标准**，否则很容易把一个大杂烩拆成多个互相调用的小杂烩。

这次改造要改成下面这个判断口径：

- 方案的最终目标是让“显示、操作、缓存、计算、元数据/规则、持久化”各自归位
- 拆文件只是手段，不是目标
- 只要没有减少状态源、分支数和排错链路长度，就不算真正完成
- 只要还需要靠按钮补刷新、Grid 补判断、下游补兜底，就说明职责还没收干净

### 0.2 必须先纠偏的 5 个点

#### 1. 不能把“整个前端共享基础设施”误判成 V2 遗留

当前文档里把 `src/hooks/`、`src/composables/` 近似等同于 V2 遗留，这是不准确的。  
实际代码里，`src/hooks/` 仍被登录页、首页、布局、菜单、store 等大量**非成本页**模块使用，不能因为 V3 成本页不用，就整体删除。

执行时要改成：

- 目标不是“整个项目删除 V2 风格目录”
- 目标是“**成本页 V3 链路**不再依赖旧目录里的实现”
- 只有确认**全项目零引用**的文件，才能物理删除

#### 2. `/dynamic` 不能一步删，必须有兼容窗口

文档里把 `/dynamic` 视为可直接删除，这个风险偏高。  
当前路由、类型声明、elegant-router 映射里仍同时存在 `dynamic` 和 `dynamic-v3`。此外数据库菜单表里很可能还有 `dynamic` 组件名。

正确做法应该是：

- 先保留 `/dynamic`
- 让 `/dynamic` 在过渡期内**重定向到** `/dynamic-v3`，或者保留一个兼容壳
- 等数据库菜单、权限配置、收藏链接都切完，再删旧入口

否则前端编译虽然能过，线上菜单仍然会 404。

#### 3. `useMetaColumns` / `useWebSocket` 不要只做“物理搬家”

文档现在写的是先从旧目录搬到新目录，后面再继续拆。这个做法不是错，但**收益偏低**，而且容易制造两次 churn。

更稳的做法是：

- `useMetaColumns`：先按“职责”拆成 `metadata shared helpers`，不要简单把整个文件平移到 `meta-v3/`
- `useWebSocket`：它本质上是**共享基础设施**，不是成本页私有逻辑。更合理的是保留底层 websocket service 在 shared 层，V3 runtime 只包一层 `hot reload adapter`

也就是说：

- “调用方切到 V3”是对的
- “底层实现一定要搬进 V3 目录”不一定对

#### 4. 第五步“响应式缓存层”方案写得过于乐观

文档把问题近似描述成：`detailCache` 不是响应式，所以 UI 不刷新；改成 `shallowReactive(new Map())` + `:row-data` 就能解决。

这个判断**不够完整**。  
结合当前代码和最近已经暴露过的问题，真实风险不只在“Vue 响应式”，还包括：

- AG Grid 对 `rowData` 的刷新依赖**引用变化和行标识稳定**
- 保存后 `row.id`、显示主键列、`_rowKey` 三者的职责必须分清
- 删除是“直接移除”还是“标红待保存”取决于**当前行真实持久化状态**，不是取决于调用时传进来的旧对象
- 后端动作执行后，主表/明细是否重载，属于**运行时刷新契约**，不是单纯缓存是否 reactive

所以第五步不能写成“改 reactive Map 即可”，必须先定义状态契约，再决定是否需要缓存层重构。

#### 5. 验证标准不能只靠 `pnpm build`

文档里每一步都强调 `pnpm build`，这个是必要的，但远远不够。  
这套成本页最近暴露的问题，大部分都是：

- 编译通过
- 页面能打开
- 但实际业务链路错了

所以每一步都必须补**行为回归用例**，否则很容易在“看起来整洁了”之后，把保存、删除、BOM 生成、同步采购价这类真业务链路改坏。

### 0.3 执行前必须冻结的 8 条约束

这 8 条建议直接视为“架构红线”，执行过程中不能破：

1. 显示层不拥有业务真相
   规则：Grid、表单、弹窗只负责显示和最小交互，不允许把 Grid API 或组件内部状态当成事实数据源。

2. 操作层只翻译用户动作，不直接碰显示细节
   规则：复制、Lookup 回填、按钮动作、单元格编辑都属于“操作”；操作层只能产生命令或 patch，不允许顺手决定 `refreshCells`、`redrawRows`、`setRowData` 这类显示行为。

3. 缓存层必须是唯一事实数据源
   规则：主表和明细都必须有明确的状态归属；不允许“主表靠 Grid、明细靠 Map、保存链再补一套临时对象”这种多真相并存。

4. 计算层必须是纯派生层
   规则：计算层只接收输入状态和变更，产出派生 patch；不直接操作 Grid，不直接决定 UI 刷新，不顺手打脏或补状态。

5. 主表和明细应共享同一条变更管线
   规则：主从可以有不同依赖传播方向，但不能走两套临时拼起来的处理流程。用户操作后，必须统一经过“写缓存 → 计算派生 → 提交 patch → 显示更新”。

6. `row.id`、显示主键列、`_rowKey` 职责必须分离
   规则：`row.id` 是内部持久化标识；显示主键列是页面字段；`_rowKey` 是稳定 UI 身份。三者允许不同，但边界必须固定，不能混用。

7. 删除语义和刷新契约必须统一
   规则：`id <= 0` 的临时行可直接移除；`id > 0` 的已落库行必须先 `_isDeleted = true`。任何影响当前页面可见数据的动作，都必须走统一刷新契约，而不是按钮里各自补刷新。

8. 兼容和兜底只能停留在边界层
   规则：历史路由兼容、后端响应兼容、第三方组件适配可以留在边界；一旦进入核心业务层，就不允许靠多层 fallback、旧状态猜测、重复同步来“让页面看起来没坏”。

### 0.3.1 本次改造的目标分层

这次改造最终要形成的是下面 6 层，而不是“更多目录、更细的文件夹”：

- 显示层：只负责渲染和最小交互
- 操作层：把用户动作翻译成命令，例如编辑、Lookup 回填、复制、删除、按钮动作
- 缓存层：保存主表、明细、选中态等唯一事实状态
- 计算层：根据状态变化生成派生字段 patch、汇总和广播结果
- 元数据/规则层：负责列、权限、校验、Lookup、公式、布局等规则解析
- 持久化层：负责查询、保存、同步后端

判断一次改造是否合格，不看“拆了几个文件”，而看下面 4 个结果：

- 代码总量是否下降，或者至少没有因为重构长期净增长
- 出问题时是否更容易定位到唯一入口
- 同一类问题是否只需要改一个地方
- 下游模块是否不再依赖上游的隐式补丁和兜底

### 0.4 建议调整后的执行顺序

比原文更稳的顺序建议是：

#### Step 0. 建基线，不改结构

- 先梳理“成本页 V3 当前必须可用”的回归清单
- 先确认哪些文件是“全项目零引用”，哪些只是“成本页零引用”
- 先确认 `/dynamic` 的数据库菜单数据是否还在使用

输出物：

- 一份“删除白名单”
- 一份“兼容保留清单”
- 一份“行为回归清单”

#### Step 1. 删除纯死代码，但只删真正零引用的

这一阶段只允许删除：

- 全项目 `rg` + 路由 +菜单数据都确认不再使用的文件
- 不能删除仍服务于登录页、布局、首页、store 的 shared hooks

#### Step 2. 先做 V3 依赖解耦，再决定是否搬目录

先把 V3 对旧目录的依赖切断：

- `useMetaConfig.ts`、`useLookupDialog.ts`、`usePageRules.ts` 不再依赖旧 `useMetaColumns`
- `MasterDetailPageV3.vue` 不再直接依赖旧 `useWebSocket`

等调用方稳定后，再决定底层文件是否物理迁移

#### Step 3. 拆 `useMetaConfig`

这一阶段只做**元数据装载链路**收口，不碰缓存策略，不碰删除/保存行为

#### Step 4. 拆 `runtime/index.ts`

这一阶段重点是：

- 初始化流程
- 热重载流程
- 动作执行后的刷新契约
- 脏标记与重算触发链

不要在这一阶段顺手改缓存实现

#### Step 5. 收口 `useMasterDetailData` 的状态契约

先明确：

- 哪个函数能改主表数据
- 哪个函数能改明细缓存
- 哪个函数负责 Grid 刷新
- 哪个函数负责保存后 ID 回填
- 哪个入口负责把“基础字段变更 → 计算派生 → patch 提交”串成统一管线

只有这一步做完，才有资格判断是否还需要“响应式缓存层”

#### Step 6. 最后再决定是否引入 `cache/`

只有在下面条件同时成立时，才建议进入原文的“第五步”：

- 现有 detail 刷新仍需要大量手工 `setGridOption('rowData', ...)`
- 状态所有权已经明确
- Grid 行标识和保存回填契约已经稳定

如果这 3 条没满足，先上 reactive cache，大概率会把问题从“刷新不稳定”升级成“状态源混乱”

### 0.5 每一步都必须回归的 8 条业务链路

不能只测“页面打开”，至少要测这 8 条：

1. 主表选择后，明细自动加载显示
2. 主表复制后，保存成功，主表主键立即转正
3. 复制后带出的明细，保存成功后，明细主键和主表外键立即转正
4. 保存成功后的主表/明细删除，先标红，再二次保存真正删除
5. “由BOM生成明细”执行后，明细无需切行/刷新即可显示
6. “同步采购价”执行后，主表/明细当前上下文即时刷新
7. Lookup 回填在主表与明细中都正常，且不会误带主键
   检查要求：回填后必须核对目标行的 `id`、`_rowKey`、显示主键列值没有被意外覆盖；尤其要覆盖“Lookup 源表和目标表都存在同名 `ID` 字段”的场景
8. WebSocket 元数据热重载后，页面不会丢失当前选中上下文

### 0.6 建议补到方案里的执行纪律

建议把下面这段当成执行纪律写进方案：

- 每一步只允许解决一个层级的问题，不允许“拆文件时顺手改行为”
- 每一步结束必须提交一次独立 commit
- 每一步结束时，必须能明确回答“当前到底是哪一套代码在生效”；如果新旧两套路径并存且没有清晰唯一 owner，则这一步视为未完成
- 每一步都要检查是否留下尾巴旧代码、重复 owner、并行逻辑；如果留下来会让后续排查时分不清“哪个在用、哪个已经废弃”，就必须在当步继续删除
- 任何涉及 `row.id`、显示主键列、`_rowKey`、`_isNew`、`_isDeleted` 的改动，都必须做复制/保存/删除回归
- 任何涉及按钮动作的改动，都必须做“执行后页面即时刷新”回归
- 如果某一步需要同时改 5 个以上核心文件，必须先拆成更小的子步骤

### 0.7 代码治理红线：简单、直接、源头收口

这部分建议直接作为本次改造的代码质量标准，否则后面很容易越改越绕。

#### 1. 能在源头解决，就不允许把问题传给下一层

例如：

- 输入数据格式不统一，就在解析层统一
- 行身份不稳定，就在状态层统一 `row.id` / `_rowKey` / 显示主键列职责
- 动作执行后要刷新，就在 runtime 刷新契约层统一，不允许每个按钮各自补丁式刷新

禁止做法：

- 上层数据已经错了，还指望下层组件自己兜住
- 明明是状态契约问题，却在 Grid 层加条件分支掩盖
- 明明是运行时刷新策略问题，却在某个按钮里单独 `refresh` 硬补

#### 2. 非必要不加兜底分支

允许的兜底只有两类：

- 外部边界不受控
  例如后端返回兼容、第三方组件异常、历史菜单入口兼容窗口
- 迁移过渡期必须短暂共存
  例如 `/dynamic` 到 `/dynamic-v3` 的兼容期

除此之外，不要随便写下面这些代码：

- 多层 `if (!xx) return fallback`
- 一层套一层的 `try/catch + 默认值`
- 同一含义维护两套状态，靠“兜底同步”保持一致
- 一个问题在 parser、runtime、grid、UI 四层都各修一遍

原则是：

- 能删掉一个分支，就不要留下它
- 能收敛成一个真相源，就不要保留两个
- 能在上游保证正确，就不要让下游猜

#### 3. 不允许为了“看起来稳”而制造隐藏复杂度

以下情况要视为负优化：

- 新增 helper 只是把复杂度藏起来，没有减少分支和职责
- 新增 adapter 只是转发参数，没有形成明确边界
- 新增中间状态只是为了补旧设计，没有真正减少耦合
- 一个函数签名没变，但内部多出大量兼容逻辑和隐式分流

判断标准不是“文件是不是变短了”，而是：

- 读代码时是否更容易知道谁负责什么
- 出问题时是否更容易定位到唯一入口
- 同一类问题是否只需要改一个地方

#### 4. 异常和兼容必须停留在边界层

真正允许写兼容逻辑的地方应该很少，通常只在：

- 路由兼容层
- 后端响应适配层
- 第三方组件适配层

一旦进入核心业务层，就应该尽量使用已经规整好的数据，不要把“历史格式兼容”“空值猜测”“多名称兜底”继续往下传。

#### 5. 每一步重构都要回答这 4 个问题

如果回答不清，就说明这一步还不够干净：

1. 这个问题的源头在哪一层
2. 为什么要在这一层修，而不是下一层兜底
3. 修完后，哪一层不再需要兼容分支
4. 这一步是否减少了状态源、分支数或调用链长度

#### 6. 每一步写完代码后，必须主动查重

这里的“查重”不是只看文件名，而是看逻辑是否重复：

- 是否出现了两套相同含义的判断
- 是否出现了两套相同含义的空值转换
- 是否出现了两套相同含义的刷新逻辑
- 是否出现了两套相同含义的状态同步逻辑

检查结论只有两种：

- 这是边界差异导致的必要重复
- 这是冗余重复，必须当场删除并收敛到一个入口

不允许出现第三种做法：

- “先留着，后面统一清”

#### 7. 新增抽象必须伴随删旧口，禁止长期叠中间层

这一条作为第二阶段及后续改造的硬规则：

- 新增一个 contract、adapter、store wrapper、helper 或中间接口后，必须在接下来的 `1~2` 步内删掉至少一层旧出口、旧分支或旧接口。
- 不允许“新接口加了，旧接口也长期留着，只是两边一起跑”。
- 如果某个抽象只是转发参数、包一层名字、没有减少职责或调用链长度，就视为无效抽象，不应继续保留。

执行检查时必须明确回答：

1. 这一步新加的抽象替换了哪个旧入口
2. 哪个旧出口已经删除，或者会在下一步删除
3. 如果暂时不能删旧口，原因是什么，最晚哪一步必须删

如果回答不清，上述抽象应视为“加料”而不是重构，需要回退或继续减量后再进入下一步。

### 1.3 数据流架构

```
useMetaConfig (元数据)
  → useMasterDetailData (主从数据 + 缓存)
      → useCalcBroadcast (计算规则广播)
          → useLookupDialog, useSave, useGridContextMenu
  → useMasterGridBindings (Grid 事件)
  → useAdvancedSearch (筛选)
  → runtime/index.ts (统一 assembly)
```

**表面优点**：无循环依赖，composable 间通过 `runtimeApi` 参数通信，整体还能跑通。

**真实问题不在“有没有拆 composable”，而在“层与层之间是否串台”**：

- 操作层和显示层耦合
  例如 Lookup 回填、按钮动作、编辑动作里，既写数据、又触发计算、又找 Grid Node、又决定刷新方式。
- 计算层和缓存层耦合
  计算逻辑不只是算值，还顺手改 row、打脏、广播、刷新 UI。
- 缓存层和显示层耦合
  明细部分有 `detailCache`，但主表事实状态仍大量依赖 Grid/SSRM 当前上下文，导致状态所有权不完整。
- 元数据/规则层和运行时编排层耦合
  元数据加载、规则编译、页面结构、动作刷新、热重载在运行时里交叉影响，导致排错要跨多个文件来回跳。

**因此当前真正的核心痛点应重写为：**

- 不是单纯“detailCache 不响应”，而是“主从状态所有权不统一”
- 不是单纯“文件太大”，而是“操作、计算、缓存、显示边界混在一起”
- 不是单纯“要不要加 reactive cache”，而是“是否已经形成唯一真相源和统一变更管线”
- 不是单纯“能不能热刷新”，而是“刷新之后是否还能保持状态和规则语义一致”

## 二、目标架构

这次改造的目标架构，不再先定义“目录应该长什么样”，而是先定义“每一层只负责什么”。

### 2.1 目标分层

#### A. 显示层

职责：

- 只负责渲染主表、明细、弹窗、表单和工具栏
- 只保留最小交互，例如点击、输入、选择、展开、关闭
- 不保存业务事实状态

禁止：

- 不直接改业务数据
- 不直接调用计算逻辑
- 不自己决定业务刷新语义

#### B. 操作层

职责：

- 把用户动作翻译成命令
- 例如：`editCell`、`lookupFillback`、`copyRow`、`deleteRow`、`runToolbarAction`

禁止：

- 不直接操作 Grid API 当成状态源
- 不顺手做计算
- 不顺手做 UI 兜底刷新

#### C. 缓存层

职责：

- 成为主表、明细、选中态、脏状态的唯一事实数据源
- 对外暴露统一的读写入口和 patch 提交机制

禁止：

- 不直接耦合具体组件
- 不把显示状态和业务状态混在一起

#### D. 计算层

职责：

- 根据缓存变化和规则生成派生 patch
- 负责公式、汇总、广播依赖传播

禁止：

- 不直接改 Grid
- 不直接决定显示刷新
- 不自己维护另一份状态

#### E. 元数据/规则层

职责：

- 负责页面结构、列定义、Lookup、权限、校验、公式、布局配置的解析与装配

禁止：

- 不混入运行时刷新逻辑
- 不混入业务状态写入逻辑

#### F. 持久化层

职责：

- 负责查询、保存、同步后端、动作接口

禁止：

- 不在持久化层补前端状态兜底

### 2.2 目标变更管线

改造完成后，主表和明细都应共享同一条处理管线：

1. 用户操作进入操作层
2. 操作层把基础字段变更写入缓存层
3. 计算层基于变更生成派生 patch
4. 缓存层统一提交基础 patch + 派生 patch
5. 显示层只订阅缓存变化并刷新

这里主从的区别只允许体现在“依赖传播方向不同”，不允许体现在“走两套不同逻辑”。

### 2.3 目录结构只是服务于分层

在这个前提下，目录可以收口成下面这种形态：

```text
meta-v3/
├── metadata/        ← 页面结构、列、权限、规则、Lookup、布局
├── runtime/         ← 只负责编排、初始化、热重载、动作路由
├── state/           ← 主表/明细/选中态/脏状态的唯一事实数据源
├── calc/            ← 公式、汇总、广播、派生 patch
├── actions/         ← 用户动作命令：编辑、复制、删除、Lookup、按钮动作
├── persistence/     ← 查询、保存、同步后端
└── view-support/    ← 只服务显示层的轻量适配
```

这不要求一次性重命名整个目录。执行时允许先沿用现有文件名和目录，但每一步都必须朝这个边界收口。

### 2.4 成功标准

本次改造完成，不以“拆出多少新文件”为标准，而以这 6 条为准：

- 代码总量明显下降，或者至少热点模块净减量
- 排错时需要跳转的文件数明显减少
- 主表和明细都拥有清晰的唯一状态入口
- 操作、计算、缓存、显示之间不再互相越层调用
- 热刷新、保存、复制、删除、Lookup 等关键链路不再依赖按钮补丁或下游兜底
- 新人阅读时能在几分钟内看出“这个问题该去哪个层查”

### 2.5 TablePanel 的目标也按同一标准执行

`TablePanel.vue` 的拆分也不是为了多几个组件，而是为了让：

- 布局容器只负责布局
- 表元数据编辑只负责表级 CRUD
- 列元数据编辑只负责列级 CRUD 和校验
- 视图列拾取弹窗只负责拾取动作

如果拆完之后仍然要跨 3 个子组件才能定位一个列配置问题，就说明拆分仍然不合格。

### 第五步：缓存实现重评估（高难度，可选，不是默认落点）

**现状痛点：**
- `detailCache` 是普通 `Map`，修改后 UI 不会自动更新
- 需要手动调用 `api.setGridOption('rowData', newRows)` 推数据
- Master SSRM + Detail Client-Side 两套模型混合

**注意：这一节不是既定终局，而是候选方向。**  
只有在主从状态契约、统一变更管线、行身份和保存回填都稳定后，才允许进入这一节。

**候选方案（仅在状态契约已收口后再落地）：**

```ts
// cache/useDetailCache.ts — 新增

// 注意：这只是候选实现，不是默认答案
// 必须先确认 Grid 行标识、保存回填、删除语义都稳定，再评估是否引入 reactive cache
const detailCache = shallowReactive(new Map<string, Record<string, RowData[]>>())

// 暴露响应式 getter
function getDetailRows(masterKey: string, tabKey: string): RowData[] {
  return detailCache.get(masterKey)?.[tabKey] ?? []
}

// 变更自动触发 UI 刷新
function setDetailRows(masterKey: string, tabKey: string, rows: RowData[]) {
  const existing = detailCache.get(masterKey) ?? {}
  existing[tabKey] = rows
  detailCache.set(masterKey, { ...existing })  // 新引用触发响应
}
```

**Detail Grid 绑定改造：**
```vue
<!-- 改造前 -->
<ag-grid-vue :row-data="getRowDataManually()" />

<!-- 改造后 -->
<ag-grid-vue :row-data="getDetailRows(currentMasterKey, currentTab)" />
<!-- computed 直接追踪 detailCache 变化，自动刷新 -->
```

**关键约束：**
- AG Grid 的 `:row-data` 只在引用变化时响应，所以 `setDetailRows` 必须创建新数组/对象引用
- Master Grid 继续用 SSRM 不变
- `useMasterDetailData.ts` 中的缓存操作全部迁移到 `useDetailCache.ts`
- 对外行为不变：增删改操作的 API 签名保持一致

**验证：**
- 明细数据加载后自动显示（无需手动 refresh）
- 行增删后 Grid 自动更新
- 计算字段广播后值自动刷新
- 切换主表行时明细缓存命中正常

## 五、改造顺序总览

这一节是“主阶段总览”。  
第六节里的 `Step 0-8` 是更细的施工拆分，其中 `Step 0` 代表基线建立，不与这里的主阶段编号冲突。

```
第一步  切断旧依赖并删除死代码      难度 ★★☆☆☆  目标：减量、去历史包袱、减少误导入口
第二步  收口元数据/规则层           难度 ★★★☆☆  目标：把页面结构、列、规则从运行时编排中剥离
第三步  收口运行时编排层           难度 ★★★☆☆  目标：让 runtime 只做路由和编排，不再混状态与计算
第四步  收口后台配置页职责         难度 ★★☆☆☆  目标：让 TablePanel 不再混布局、CRUD、校验、弹窗
第五步  收口主从状态契约           难度 ★★★★☆  目标：建立唯一事实数据源，统一主从变更管线
第六步  最后评估缓存实现是否重做   难度 ★★★★★  目标：只有在状态契约稳定后，才决定是否引入新的 cache 层
```

每一步独立，做完一步验证通过再做下一步。  
成功标准不是“新文件变多了”，而是“代码变少了、边界变清楚了、排错路径变短了”。

## 六、逐步施工单（强制由简单到困难）

这一节不是原则描述，而是**实际执行顺序**。  
执行时必须按顺序推进，前一步没过，不允许进入下一步。

### 6.0 先看这张表：难度、风险、准入门槛

| Step | 主题 | 难度 | 页面崩溃风险 | 进入条件 | 本步失败时的回退方式 |
|------|------|------|--------------|----------|----------------------|
| 0 | 建立基线 | ★☆☆☆☆ | 很低 | 无 | 直接保留文档，不动代码 |
| 1 | 旧依赖解耦 | ★★☆☆☆ | 低 | 已确认 V3 当前旧依赖点 | 回退 import 切换和 adapter 新文件 |

执行纪律：

- 同一时间只允许存在一个 `in progress` 的 Step
- Step 4 以后，每一步都必须先在一个典型成本页试跑，再推广
- Step 7 和 Step 8 不允许和“删旧文件”一起做

### 6.1 通用施工规则

#### 单步边界

- 每一步只允许处理**一个功能模块**
- 单步尽量控制在 `1~3` 个核心文件；纯 import 切换、类型搬迁可放宽
- 单步禁止同时做“结构拆分 + 行为修改 + 样式调整”
- 单步如果已经改到保存链、删除链、热重载链中的任意一条，就不允许再顺手改别的模块
- 单步不能只“把代码搬家”，必须明确回答这一步到底减少了哪个层级的职责混杂

#### 单步输出物

每一步结束必须产出：

- 一个独立 commit
- 一份“本步修改文件清单”
- 一份“本步已回归用例清单”
- 一份“未处理风险清单”
- 一份“本步净减量说明”
  至少说明：减少了哪些分支、减少了哪些状态源、减少了哪些跨层调用
- 一份“重复逻辑检查结果”
  至少说明：本步新增代码里是否出现了重复判断、重复转换、重复刷新、重复状态同步；如果是冗余重复，必须在本步直接删除
- 一份“新旧接口替换结果”
  至少说明：本步新增的抽象替换了哪些旧出口；若存在暂未删除的旧口，必须写明最晚删除步骤
- 一份“下一步是否允许进入”的结论

#### 单步测试门禁

每一步改造完成后，必须先测试，再决定是否进入下一步。  
本轮改造以**前端重构**为主，所以默认策略改为：

- 不要求每一步都重启后端
- 只要没有改到接口契约、鉴权、代理、WebSocket、环境变量，就保持当前后端运行状态即可
- 每一步优先验证“本步涉及的前端模块”和“相邻核心链路”是否正常

只有在下面情况出现时，才需要升级为更完整的联调验证：

- 改了请求参数/响应结构
- 改了登录、token、权限、路由守卫
- 改了 Vite 代理、服务地址、环境变量
- 改了 WebSocket 连接、消息订阅、热重载链路

这里的“测试通过”不能只指 `pnpm build` 通过，而是至少同时满足下面 4 条：

1. 静态检查通过
   包括本步涉及文件的 lint / typecheck / build，不允许新增当前步骤引入的错误
2. 前端页面能正常打开
   至少要进入一个真实成本页，确认没有白屏、路由异常、初始化失败
3. 本步相关模块回归通过
   例如本步改到元数据，就回归列加载、Lookup、规则编译；改到 runtime，就回归热重载、脏标记、动作刷新
4. 相邻核心链路没有被顺带改坏
   至少补测与本步最接近的基础链路，不要求每一步都把全业务重新走一遍

只要上面任一项不过，本步就视为**未完成**，不得进入下一步。

#### 单步停机线

只要出现下面任一情况，本步立即停止，不继续扩散：

- 主表不能正常加载
- 明细不能随主表切换正常显示
- 复制/保存/删除链路出现语义回退
- 按钮动作执行后页面不刷新
- 热重载导致当前页面上下文丢失

出现停机线后的处理顺序必须固定：

1. 先停止继续加需求
2. 先把当前问题定位并修完
3. 重新完整回归本步测试清单
4. 只有重新通过后，才允许继续下一步

### 6.6 Step 4：拆 `useMetaConfig`，但只做委托拆分

#### 目标

把元数据装载链从“大函数”拆成多个子模块，但对外 API 完全不变。

#### 本步允许做的事

- 提取 `parseConfig`
- 提取 `loadMeta`
- 提取 `compileRules`

#### 本步禁止做的事

- 不改返回结构
- 不改 runtime 调用方式
- 不在这一阶段改规则语义

#### 建议子步骤

1. 先提 `useRuleCompiler.ts`
   原因：`compileRules()` 位于文件底部，职责最集中，最适合作为第一块独立提取
2. 再提 `useComponentLoader.ts`
   原因：`loadComponents()` 逻辑最短、依赖最浅，适合作为第二块低风险提取
3. 再提 `useConfigParser.ts`

每提一个子模块就回归一次，不要一口气拆完。

1. 只提取一个职责块
2. 保持原导出签名不变

### 6.9 Step 7：先收口主从状态契约，再决定是否重做缓存层

#### 目标

先把“谁维护真相”和“主从是否共用同一条变更管线”讲清楚，再谈 reactive cache。

#### 本步必须收口的 4 类语义

1. 行身份
   `row.id`、显示主键列、`_rowKey` 的职责边界
2. 删除语义
   什么情况下直接移除，什么情况下标红待保存
3. 保存语义
   保存成功后主表/明细主键如何回填
4. 刷新语义
   主表、明细、按钮动作执行后的刷新粒度

#### 本步必须形成的统一管线

无论主表还是明细，都必须尽量收敛成下面这条顺序：

1. 用户动作进入操作层
2. 基础字段变化先进入缓存层
3. 计算层基于变更生成派生 patch
4. 缓存层统一提交 patch
5. 显示层订阅变化并刷新

允许不同的只有依赖传播方向，不允许不同的则是处理流程本身。

#### 本步禁止做的事

- 不先上 `cache/`
- 不先上 reactive Map
- 不在状态约束没稳定前重写 detail 数据源
- 不和 Step 8 合并提交

#### 完成标准

- 复制后保存，主表/明细主键立即转正
- 保存后删除，先标红再二次保存删除
- BOM / 同步采购价 执行后页面即时刷新
- 删除时“直接消失”和“标红待保存”的边界只由真实持久化状态决定，不由旧对象残留状态决定
- 主表和明细的编辑、Lookup、复制、删除不再依赖两套不同的临时流程

### 6.10 Step 8：最后才评估响应式缓存层

#### 进入条件

- 现有刷新逻辑仍然分散在多个模块里
- `row.id` / `_rowKey` / 删除语义 / 保存回填都已经稳定
- 明细 UI 刷新仍然严重依赖手工 `setGridOption('rowData', ...)`

#### 本步推荐做法

- 先新增 `cache/` 层
- 先只接入一个明细 tab 做试点
- 试点稳定后再推广到全部 tab
- 试点期间保留旧路径可切回，不做一次性替换

#### 本步禁止做的事

- 不一次性替换所有 detail 读写路径
- 不把 master SSRM 一起重写

#### 完成标准

- 试点 tab 自动刷新稳定
- 主从切换不丢缓存
- 不引入新的身份错乱、删除错判、保存回填异常
- 一轮真实业务试点后，再决定是否推广到全部明细 tab

### 6.12 每一步结束后的固定动作

每一步结束后，必须严格按下面顺序执行：

1. 自查本步改动范围是否超界
2. 跑本步静态检查
3. 在当前后端运行状态下，跑本步相关前端模块回归
4. 补测相邻核心链路，确认没有连带回归
5. 记录问题和残留风险
6. 只有确认“本步通过”，才允许提交并进入下一步

补充规则：

- 如果本步只是前端内部重构，不要求重启后端
- 如果本步触到了接口契约、鉴权、代理、WebSocket，再升级为联调验证

如果第 2 到第 4 步里任何一项失败：

- 不允许带着问题进入下一步
- 不允许把问题留到“后面统一修”
- 必须在当前步骤内收口，或者直接回退当前步骤

### 6.13 最后执行原则

这份方案真正要避免的不是“代码改多了”这一句口号，而是下面 4 种失败结果：

- 文件拆散了，但代码总量没降，甚至长期净增长
- 大文件没了，但职责还是混的，只是换了地方继续串台
- 下游显示层、Grid 层、按钮层到处补判断，靠兜底把问题压住
- 一次改动后仍然要跨 5 个以上文件才能定位问题

所以执行时必须遵守一句话：

**先收边界，再减代码；先收真相源，再谈响应式；先统一管线，再优化实现。**

只要某一步做完后还需要用下面这些话解释它：

- “这里先加一层兼容，后面再收”
- “这里先保留旧状态，后面再统一”
- “这里先补个刷新，避免页面不动”

那这一步就大概率还没有达到本方案的质量标准。

### 6.14 暂缓项：权限变更后的前端热刷新链路

当前已确认：

- 元数据配置、列覆盖配置变更后，后端会通过 WebSocket 广播 `META_CONFIG_CHANGED`，业务页可热刷新。
- 权限管理里的页面列权限、按钮权限、行权限变更后，后端当前没有补同类广播，因此业务页通常需要手动刷新后才会重新拉取最新权限结果。

这不是本轮前端结构重构的阻塞项，但必须明确记录：

- “权限变更后通知前端热刷”这条链当前还没做完整。
- 本次改造计划执行期间，不把它混入前端模块拆分步骤里顺手处理，避免把结构重构和权限刷新链路耦在一起。
- 等本份改造计划全部完成、V3 结构稳定后，再单独立项补这条链路。

后续单独处理时，目标应明确为：

1. 权限变更保存成功后，后端主动广播对应页面变更事件。
2. 前端业务页收到事件后，按权限类型执行最小必要刷新：
   列权限重载 metadata。
   按钮权限重载 page components。
   行权限重拉当前数据。
3. 权限的“只收紧、不放大”语义不变，只补通知和热刷新链，不顺手改权限合并规则。

本计划全部完成后，需要专门通知并安排这条链路的实现与回归测试。
