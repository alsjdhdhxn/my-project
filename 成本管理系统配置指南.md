## 1. 配置总览与流程

### 1.1 配置一个新页面的步骤

```
步骤1: 创建数据库表/视图/序列（业务表）
步骤2: 配置 T_COST_TABLE_METADATA（注册表）
步骤3: 配置 T_COST_COLUMN_METADATA（注册列）
步骤4: 配置 T_COST_PAGE_COMPONENT（定义页面结构）
步骤5: 配置 T_COST_PAGE_RULE（定义页面行为）
步骤6: 配置 T_COST_LOOKUP_CONFIG（如需弹窗选择器）
步骤7: 配置 T_COST_EXPORT_CONFIG（如需自定义导出）
步骤8: 配置 T_COST_ROLE_PAGE（如需权限控制）
步骤9: 前端路由注册 pageCode
```

### 1.2 命名规范速查

| 对象 | 规范 | 示例 |
|------|------|------|
| 数据库表名 | `T_COST_大写下划线` | `T_COST_PINGGU` |
| 数据库视图 | `V_COST_大写下划线` | `V_COST_PINGGU_MATERIAL` |
| 序列 | `SEQ_大写下划线` | `SEQ_COST_PINGGU` |
| TABLE_CODE | 大驼峰 | `CostPinggu` |
| FIELD_NAME | 小驼峰 | `costDate`、`outPriceRmb` |
| COLUMN_NAME | 大写下划线 | `COST_DATE`、`OUT_PRICE_RMB` |
| PAGE_CODE | 小写中划线 | `cost-pinggu`、`wms-shipping` |
| COMPONENT_KEY | 小驼峰 | `masterGrid`、`detailTabs` |

### 1.3 V3 强制约束

- 主表 Grid 强制使用 Server-Side Row Model (SSRM)，无论 GRID_OPTIONS 怎么配
- 明细表 Grid 强制使用 Client-Side Row Model
- 所有业务数据走 DynamicMapper，禁止创建独立 Controller/Service/Mapper

## 2. 表元数据

### 2.1 表结构 T_COST_TABLE_METADATA

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ID | NUMBER | 是 | 主键，使用 SEQ_COST_TABLE_METADATA |
| TABLE_CODE | VARCHAR2 | 是 | 表编码，大驼峰，全局唯一 |
| TABLE_NAME | VARCHAR2 | 是 | 表中文名称 |
| QUERY_VIEW | VARCHAR2 | 是 | 查询数据源（表名或视图名），用于 SELECT |
| TARGET_TABLE | VARCHAR2 | 是 | 保存目标表，用于 INSERT/UPDATE/DELETE |
| SEQUENCE_NAME | VARCHAR2 | 否 | 序列名，新增时自动获取 ID |
| PK_COLUMN | VARCHAR2 | 是 | 主键列名（数据库列名），如 `ID`、`DOCID` |
| PARENT_TABLE_CODE | VARCHAR2 | 否 | 父表 TABLE_CODE（从表才需要） |
| PARENT_FK_COLUMN | VARCHAR2 | 否 | 关联父表的外键列名（数据库列名） |
| AUDIT_ENABLED | NUMBER(1) | 否 | 是否启用审计追踪，0=否 1=是 |
| VALIDATION_RULES | CLOB | 否 | 后端验证规则 JSON（可选） |
| ACTION_RULES | CLOB | 否 | 后端动作规则 JSON（可选） |
| DESCRIPTION | VARCHAR2 | 否 | 表描述说明 |
| DELETED | NUMBER(1) | 是 | 软删除标记，0=正常 |

### 2.2 配置示例

#### 单表（无主从关系）

```sql
INSERT INTO T_COST_TABLE_METADATA (ID, TABLE_CODE, TABLE_NAME, QUERY_VIEW, TARGET_TABLE, SEQUENCE_NAME, PK_COLUMN, DELETED, CREATE_TIME, UPDATE_TIME, CREATE_BY)
VALUES (5, 'CostUser', '用户管理', 'V_COST_USER', 'T_COST_USER', 'SEQ_COST_USER', 'ID', 0, SYSTIMESTAMP, SYSTIMESTAMP, 'system');
```

#### 主从表（主表）

```sql
-- 主表：PARENT_TABLE_CODE 和 PARENT_FK_COLUMN 留空
INSERT INTO T_COST_TABLE_METADATA (ID, TABLE_CODE, TABLE_NAME, QUERY_VIEW, TARGET_TABLE, SEQUENCE_NAME, PK_COLUMN, PARENT_TABLE_CODE, PARENT_FK_COLUMN, DELETED, CREATE_TIME, UPDATE_TIME, CREATE_BY)
VALUES (121, 'CostPinggu', '物料成本核算表', 'T_COST_PINGGU', 'T_COST_PINGGU', 'SEQ_COST_PINGGU', 'DOCID', NULL, NULL, 0, SYSTIMESTAMP, SYSTIMESTAMP, 'system');
```

#### 主从表（从表）

```sql
-- 从表：PARENT_TABLE_CODE 指向主表，PARENT_FK_COLUMN 指定外键列
INSERT INTO T_COST_TABLE_METADATA (ID, TABLE_CODE, TABLE_NAME, QUERY_VIEW, TARGET_TABLE, SEQUENCE_NAME, PK_COLUMN, PARENT_TABLE_CODE, PARENT_FK_COLUMN, DELETED, CREATE_TIME, UPDATE_TIME, CREATE_BY)
VALUES (122, 'CostMaterial', '原辅料明细', 'V_COST_PINGGU_MATERIAL', 'T_COST_PINGGU_DTL', 'SEQ_COST_PINGGU_DTL', 'DTLID', 'CostPinggu', 'DOCID', 0, SYSTIMESTAMP, SYSTIMESTAMP, 'system');
```

### 2.3 关键说明

- `QUERY_VIEW` 和 `TARGET_TABLE` 可以不同：查询用视图（可关联多表），保存用物理表
- 多个从表可以共用同一个 `TARGET_TABLE`（如原料和包材都存 `T_COST_PINGGU_DTL`），通过分组字段区分
- `PK_COLUMN` 是数据库列名（大写），不是前端字段名
- `PARENT_FK_COLUMN` 是从表中关联主表主键的数据库列名

## 3. 列元数据

### 3.1 表结构 T_COST_COLUMN_METADATA

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ID | NUMBER | 是 | 主键 |
| TABLE_METADATA_ID | NUMBER | 是 | 关联 T_COST_TABLE_METADATA.ID |
| FIELD_NAME | VARCHAR2 | 是 | 前端字段名（小驼峰） |
| COLUMN_NAME | VARCHAR2 | 是 | 默认数据库列名（大写下划线） |
| QUERY_COLUMN | VARCHAR2 | 否 | 查询列名（视图中的列名），为空时用 COLUMN_NAME |
| TARGET_COLUMN | VARCHAR2 | 否 | 保存列名（物理表中的列名），为空时用 COLUMN_NAME |
| HEADER_TEXT | VARCHAR2 | 是 | 列标题（中文） |
| DATA_TYPE | VARCHAR2 | 是 | 数据类型：`text`/`number`/`date`/`datetime`/`select` |
| DISPLAY_ORDER | NUMBER | 是 | 显示顺序，从 0 开始 |
| SORTABLE | NUMBER(1) | 否 | 是否可排序，0=否 1=是 |
| FILTERABLE | NUMBER(1) | 否 | 是否可筛选，0=否 1=是 |
| DICT_TYPE | VARCHAR2 | 否 | 字典类型编码（用于下拉选择） |
| IS_VIRTUAL | NUMBER(1) | 否 | 是否虚拟列，0=物理列 1=虚拟列（不存不取，仅公式计算） |
| DELETED | NUMBER(1) | 是 | 软删除标记 |

### 3.2 QUERY_COLUMN 与 TARGET_COLUMN 的用途

当查询视图和保存表的列名不同时使用：

```sql
-- 视图中列名是 MASTER_ID，物理表中列名是 DOCID
INSERT INTO T_COST_COLUMN_METADATA (..., FIELD_NAME, COLUMN_NAME, QUERY_COLUMN, TARGET_COLUMN, ...)
VALUES (..., 'masterId', 'MASTER_ID', 'MASTER_ID', 'DOCID', ...);
```

- 查询时用 `QUERY_COLUMN`（或 `COLUMN_NAME`）
- 保存时用 `TARGET_COLUMN`（或 `COLUMN_NAME`）
- 前端统一用 `FIELD_NAME`

### 3.3 虚拟列（IS_VIRTUAL = 1）

虚拟列不会从数据库查询，也不会保存到数据库，仅用于前端公式计算的中间值或展示。

```sql
-- formulaType 是虚拟列，仅用于前端多公式选择
INSERT INTO T_COST_COLUMN_METADATA (..., FIELD_NAME, COLUMN_NAME, HEADER_TEXT, DATA_TYPE, IS_VIRTUAL, ...)
VALUES (..., 'formulaType', 'FORMULA_TYPE', '公式类型', 'text', 1, ...);
```

### 3.4 DATA_TYPE 取值

| 值 | 说明 | AG Grid 效果 |
|----|------|-------------|
| `text` | 文本 | 文本筛选器 |
| `number` | 数字 | 数字筛选器，右对齐 |
| `date` | 日期 | 日期筛选器 |
| `datetime` | 日期时间 | 日期筛选器 |
| `select` | 下拉选择 | 配合 DICT_TYPE 使用 |

### 3.5 配置示例

```sql
-- 主键列（通常隐藏）
INSERT INTO T_COST_COLUMN_METADATA (ID, TABLE_METADATA_ID, FIELD_NAME, COLUMN_NAME, HEADER_TEXT, DATA_TYPE, DISPLAY_ORDER, SORTABLE, FILTERABLE, IS_VIRTUAL, DELETED, CREATE_TIME, UPDATE_TIME, CREATE_BY)
VALUES (1313, 121, 'id', 'DOCID', 'ID', 'number', 0, 1, 1, 0, 0, SYSTIMESTAMP, SYSTIMESTAMP, 'system');

-- 普通文本列
VALUES (1314, 121, 'goodsname', 'GOODSNAME', '产品名称', 'text', 1, 1, 1, 0, 0, SYSTIMESTAMP, SYSTIMESTAMP, 'system');

-- 数字列
VALUES (1320, 121, 'apexPl', 'APEX_PL', '批量', 'number', 7, 1, 1, 0, 0, SYSTIMESTAMP, SYSTIMESTAMP, 'system');

-- 从表外键列（QUERY_COLUMN 和 TARGET_COLUMN 不同）
VALUES (1343, 122, 'masterId', 'MASTER_ID', 'MASTER_ID', 'DOCID', '主表ID', 'number', 1, 1, 1, 0, 0, SYSTIMESTAMP, SYSTIMESTAMP, 'system', 'CostMaterial');
```

### 4.4 GRID 组件配置

GRID 组件的 `COMPONENT_CONFIG` 主要用于配置按钮：

```json
{
  "height": "50%",
  "selectionMode": "single",
  "buttons": [
    {"action": "addRow", "label": "新增"},
    {"action": "copyRow", "label": "复制", "requiresRow": true},
    {"action": "deleteRow", "label": "删除", "requiresRow": true},
    {"action": "save", "label": "保存"},
    {"action": "saveGridConfig", "label": "保存列配置"},
    {"action": "clipboard.copy", "label": "复制"},
    {"action": "clipboard.paste", "label": "粘贴"}
  ]
}
```

#### 按钮配置详解

每个按钮项的完整字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| action | string | 动作标识（见下方 action 列表） |
| label | string | 按钮文字 |
| type | string | `action`（默认）/ `separator`（分隔线）/ `group`（子菜单） |
| position | string | 显示位置：`context`（右键菜单，默认）/ `toolbar`（工具栏）/ `both`（两处都显示） |
| requiresRow | boolean | 是否需要选中行才能操作 |
| requiresSelection | boolean | 是否需要有选中数据 |
| disabled | boolean | 是否禁用 |
| visible | boolean | 是否可见 |
| confirm | string | 确认提示文字（点击后弹确认框） |
| buttonType | string | 工具栏按钮样式：`default`/`primary`/`info`/`success`/`warning`/`error` |
| permission | string | 权限标识 |
| refreshMode | string | 执行后刷新模式：`all`/`row`/`detail`/`none` |
| items | array | 子菜单项（type=group 时使用） |

#### 内置 action 列表

| action | 说明 | 备注 |
|--------|------|------|
| `addRow` | 新增行 | |
| `copyRow` | 复制行 | 需要 requiresRow |
| `deleteRow` | 删除行 | 需要 requiresRow |
| `save` | 保存 | |
| `saveGridConfig` | 保存列配置 | |
| `clipboard.copy` | 复制到剪贴板 | |
| `clipboard.paste` | 从剪贴板粘贴 | |
| `batchSelect` | 批量选择 | 需要额外配置 batchSelectConfig |

#### 后端 action（存储过程调用）

非内置 action 会自动调用后端执行接口。可配置存储过程：

```json
{
  "action": "generateFromBom",
  "label": "由BOM生成明细",
  "requiresRow": true,
  "procedure": "P_COST_BOM_INSERT",
  "params": [
    {"source": "data.id", "mode": "IN", "jdbcType": "NUMERIC"}
  ],
  "refreshMode": "detail"
}
```

| 字段 | 说明 |
|------|------|
| procedure | 存储过程名称 |
| params | 参数列表 |
| params[].source | 参数来源：`data.字段名`（从当前行取值） |
| params[].mode | 参数模式：`IN`/`OUT`/`INOUT` |
| params[].jdbcType | JDBC 类型：`NUMERIC`/`VARCHAR` 等 |
| refreshMode | 执行后刷新：`all`=刷新全部，`row`=刷新当前行，`detail`=刷新明细，`none`=不刷新 |

#### 批量选择 action 配置

```json
{
  "action": "batchSelect",
  "label": "从库存添加",
  "batchSelectConfig": {
    "lookupCode": "wmsQtyLookup",
    "title": "选择库存",
    "mapping": {
      "goodsid": "goodsid",
      "goodsgp": "goodsgp",
      "lotno": "lotno"
    },
    "filterField": "goodsid",
    "filterColumn": "GOODSID"
  }
}
```

### 4.5 TABS 组件配置

TABS 组件的 `COMPONENT_CONFIG` 定义明细表标签页：

```json
{
  "mode": "multi",
  "tabs": [
    {
      "key": "material",
      "title": "原料/辅料",
      "tableCode": "CostMaterial",
      "columns": [],
      "variantKey": null,
      "initialSort": [{"colId": "dtlUseflag", "sort": "asc"}],
      "buttons": [
        {"action": "addRow", "label": "新增"},
        {"action": "deleteRow", "label": "删除", "requiresRow": true},
        {"action": "save", "label": "保存"}
      ]
    },
    {
      "key": "package",
      "title": "包材",
      "tableCode": "CostPackage",
      "buttons": [...]
    }
  ]
}
```

#### TABS mode 说明

| mode | 说明 | 适用场景 |
|------|------|----------|
| `multi` | 多表模式，每个 Tab 对应不同的 TABLE_CODE | 从表是不同的表 |
| `group` | 分组模式，所有 Tab 共用同一个表，按字段值分组 | 从表是同一个表，按类型分组 |

#### group 模式配置

```json
{
  "mode": "group",
  "groupField": "dtlUseflag",
  "tabs": [
    {"key": "material", "title": "原料", "value": "原料", "columns": ["apexGoodsname", "price", "batchQty"]},
    {"key": "auxiliary", "title": "辅料", "value": "辅料", "columns": ["apexGoodsname", "price", "batchQty"]},
    {"key": "package", "title": "包材", "values": ["内包材", "外包材"], "columns": ["apexGoodsname", "suqty", "price"]}
  ]
}
```

- `groupField`：分组字段名
- `value`：单个分组值
- `values`：多个分组值（一个 Tab 显示多种类型）
- `columns`：该 Tab 显示的列（为空则显示全部列）

#### Tab 级配置项

| 字段 | 类型 | 说明 |
|------|------|------|
| key | string | Tab 唯一标识，也用作 componentKey 关联规则 |
| title | string | Tab 标题 |
| tableCode | string | 关联的 TABLE_CODE（multi 模式必填） |
| value | string | 分组值（group 模式） |
| values | string[] | 多个分组值（group 模式） |
| columns | string[] | 显示的列（fieldName 列表），为空显示全部 |
| initialSort | array | 初始排序：`[{"colId": "fieldName", "sort": "asc"}]` |
| variantKey | string | 变体列分组键（用于同一表不同 Tab 显示不同列） |
| buttons | array | 该 Tab 的按钮配置（格式同 GRID 的 buttons） |

## 5. 页面规则

### 5.1 表结构 T_COST_PAGE_RULE

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| ID | NUMBER | 是 | 主键 |
| PAGE_CODE | VARCHAR2 | 是 | 页面编码 |
| COMPONENT_KEY | VARCHAR2 | 是 | 关联的组件 KEY |
| RULE_TYPE | VARCHAR2 | 是 | 规则类型（见下表） |
| RULES | CLOB | 是 | 规则内容（JSON） |
| SORT_ORDER | NUMBER | 否 | 排序（同类型多条规则时） |
| DESCRIPTION | VARCHAR2 | 否 | 规则描述 |
| DELETED | NUMBER(1) | 是 | 软删除标记 |

### 5.2 RULE_TYPE 总览

| RULE_TYPE | 说明 | RULES 格式 | 适用组件 |
|-----------|------|-----------|----------|
| `COLUMN_OVERRIDE` | 列覆盖 | JSON 数组 | GRID |
| `CALC` | 行级计算 | JSON 数组 | GRID |
| `AGGREGATE` | 聚合计算 | JSON 数组 | masterGrid |
| `VALIDATION` | 验证规则 | JSON 数组 | GRID |
| `LOOKUP` | 弹窗选择器 | JSON 数组 | GRID |
| `BROADCAST` | 广播字段 | JSON 数组 | detailTabs |
| `GRID_OPTIONS` | Grid 扩展配置 | JSON 对象 | GRID |
| `ROW_EDITABLE` | 行级可编辑 | JSON 数组 | GRID |
| `CELL_EDITABLE` | 单元格级可编辑 | JSON 数组 | GRID |
| `ROW_CLASS` | 行级样式 | JSON 数组 | GRID |
| `CONTEXT_MENU` | 右键菜单 | JSON 对象 | GRID |
| `BUTTON` | 按钮配置 | JSON 对象/数组 | GRID（兼容旧数据） |
| `ROLE_BINDING` | 角色绑定 | JSON 对象 | 任意 |
| `RELATION` | 关联配置 | JSON 对象 | 布局 |
| `SUMMARY_CONFIG` | 汇总配置 | JSON 对象 | detailTabs |

### 5.3 COLUMN_OVERRIDE — 列覆盖

在页面级别覆盖列元数据的默认属性。


### 5.4 CALC — 行级计算

当指定字段值变化时，自动计算目标字段的值。

```sql
INSERT INTO T_COST_PAGE_RULE (..., RULE_TYPE, RULES, ...)
VALUES (..., 'CALC', '[
  {
    "field": "salemoney",
    "expression": "outPriceRmb / pPerpack * apexPl * (yield / 100)",
    "triggerFields": ["outPriceRmb", "pPerpack", "apexPl", "yield"]
  },
  {
    "field": "jgfBatch",
    "expression": "salemoney - totalCost",
    "triggerFields": ["salemoney", "totalCost"]
  }
]', ...);
```

#### 单公式模式

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| field | string | 是 | 计算结果写入的字段 |
| expression | string | 是 | 数学表达式（使用 mathjs 语法） |
| triggerFields | string[] | 是 | 触发计算的字段列表 |
| condition | string | 否 | JS 条件表达式，如 `"dtlUseflag !== '包材'"` |
| order | number | 否 | 执行顺序（支持级联计算） |

#### 多公式模式

根据某个字段的值选择不同的计算公式：

```json
[
  {
    "field": "batchQty",
    "formulaField": "formulaType",
    "formulas": {
      "原料": {
        "expression": "perHl * apexPl / 1000000 / (yield / 100) + exaddMater",
        "triggerFields": ["perHl", "apexPl", "yield", "exaddMater"]
      },
      "辅料": {
        "expression": "perHl * apexPl / 1000000",
        "triggerFields": ["perHl", "apexPl"]
      }
    },
    "triggerFields": ["formulaType", "perHl", "apexPl", "yield", "exaddMater"]
  }
]
```

| 字段 | 说明 |
|------|------|
| formulaField | 用哪个字段的值来选择公式 |
| formulas | 公式映射表，key 是 formulaField 的值，value 是公式定义 |

#### 表达式语法

- 使用 [mathjs](https://mathjs.org/) 语法
- 支持四则运算：`+` `-` `*` `/`
- 支持括号：`(a + b) * c`
- 支持函数：`sqrt(x)`、`abs(x)`、`max(a, b)` 等
- 变量名就是 fieldName
- null/undefined 自动转为 0

### 5.5 AGGREGATE — 聚合计算

从表数据汇总到主表字段。

```sql
INSERT INTO T_COST_PAGE_RULE (..., RULE_TYPE, RULES, ...)
VALUES (..., 'AGGREGATE', '[
  {"sourceField": "costBatch", "targetField": "totalYl", "algorithm": "SUM", "filter": "dtlUseflag === ''原料''"},
  {"sourceField": "costBatch", "targetField": "totalFl", "algorithm": "SUM", "filter": "dtlUseflag === ''辅料''"},
  {"sourceField": "costBatch", "targetField": "totalBc", "algorithm": "SUM"},
  {"targetField": "totalCost", "expression": "totalYl + totalFl + totalBc"}
]', ...);
```

#### 聚合规则字段

| 字段 | 类型 | 说明 |
|------|------|------|
| sourceField | string | 从表源字段（algorithm 模式必填） |
| targetField | string | 主表目标字段 |
| algorithm | string | 聚合算法：`SUM`/`AVG`/`COUNT`/`MAX`/`MIN` |
| filter | string | JS 过滤条件，如 `"dtlUseflag === '原料'"` |
| expression | string | 表达式计算（依赖其他聚合结果），如 `"totalYl + totalFl"` |

#### 执行顺序

1. 先执行有 `algorithm` 的规则（SUM/AVG 等）
2. 再执行有 `expression` 的规则（可引用第一步的结果）
3. 聚合完成后触发主表 CALC 规则（级联计算）

```sql

  {
    "field": "goodsname",
    "lookupCode": "costGoods",
    "mapping": {
      "goodsid": "goodsid",
      "goodsname": "goodsname",

      "customid": "customid",

      "fmname": "fmname",
```

#### Lookup 规则字段

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| field / fieldName | string | 是 | 点击哪个字段触发弹窗 |

### 5.10 ROW_EDITABLE — 行级可编辑规则

根据行数据条件控制整行是否可编辑。

```sql
INSERT INTO T_COST_PAGE_RULE (..., RULE_TYPE, RULES, ...)
VALUES (..., 'ROW_EDITABLE', '[
  {"field": "status", "operator": "ne", "value": "已审核"}
]', ...);
```

含义：status 不等于"已审核"的行才可编辑。所有条件必须同时满足（AND 关系）。

| 字段 | 类型 | 说明 |

| value | any | 比较值（`in`/`notIn` 时为数组） |

### 5.11 CELL_EDITABLE — 单元格级可编辑规则

比 ROW_EDITABLE 更细粒度，条件匹配时只允许编辑指定字段。

```sql
INSERT INTO T_COST_PAGE_RULE (..., RULE_TYPE, RULES, ...)
VALUES (..., 'CELL_EDITABLE', '[
  {
    "condition": {"field": "status", "operator": "eq", "value": "部分审核"},
    "editableFields": ["memo", "remark"]
  }
]', ...);
```

含义：当 status="部分审核" 时，该行只有 memo 和 remark 可编辑。

优先级：CELL_EDITABLE > ROW_EDITABLE（CELL_EDITABLE 匹配时优先使用）。

## 8. 权限配置


- 角色权限在基础元数据范围内生效（基础隐藏/不可编辑不能被放开）
- 用户偏好只能收紧可见/可编辑

| GRID_KEY | VARCHAR2 | Grid 标识（如 `masterGrid`、`material`） |
| CONFIG_DATA | CLOB | 列配置 JSON |
| DELETED | NUMBER(1) | 软删除标记 |
| UPDATE_TIME | TIMESTAMP | 更新时间 |

### 9.2 CONFIG_DATA 格式

```json
{
  "columns": [
    {"field": "goodsname", "width": 200, "order": 0, "hidden": false, "pinned": null},
    {"field": "apexPl", "width": 100, "order": 1, "hidden": false, "pinned": "left"},
    {"field": "memo", "width": 150, "order": 10, "hidden": true, "pinned": null}
  ]
}
```

### 9.3 保存方式

用户通过右键菜单"保存列配置"触发保存，自动记录当前列宽、顺序、显隐、冻结状态。

## 附录 C：内部字段约定

行数据中以 `_` 开头的字段为系统内部字段，不会保存到数据库：

| 字段 | 说明 |
|------|------|
| `_rowKey` | 行唯一标识（前端生成） |
| `_isNew` | 是否新增行 |
| `_isDeleted` | 是否已删除（软删除标记） |
| `_dirtyFields` | 脏字段追踪 `{字段名: {originalValue, type}}` |
| `_tabKey` | 汇总行所属 Tab |
| `_masterId` | 汇总行关联的主表 ID |
| `_masterRowKey` | 汇总行关联的主表行 KEY |
| `_detailRows` | 汇总行包含的明细行数据 |
